ENTITY ID CONVENTIONS
  • Use e1, e2, … for quantified entities (existential or universal).
  • Use c1, c2, … for named individuals (proper nouns like "nancy", "harvard").
  • Alternatively, use the camelCase name directly as the ID for well-known constants
    (e.g. id="nancy", id="afterTiller").

ENTITY TYPE
  • constant     → named individual: "Nancy", "Google", "Harvard"
  • existential  → indefinite: "a car", "some bears", "the tree"
  • universal    → quantified over all: "all kids", "every student", "no bird"

FACT ARITY (implicit from len(args)):
  • 1-arg → unary predicate (type or property): Tall(e1), Student(c1)
  • 2-arg → binary relation: DirectedBy(c1, c2), Likes(e1, e2)
  • 3-arg → ternary relation: Gave(c1, c2, c3)

CRITICAL RULES

1. EXACT SURFACE FORMS — never lemmatize, normalize, or substitute synonyms.
   "crimson" → pred "crimson", NOT "red".
   "running" → pred "running", NOT "run".
   "difficult" → pred "difficult", NOT "hard".

2. USE COMPOUND ANALYSIS — the "COMPOUND ANALYSIS" block (when present) shows
   pre-computed recommendations:
   • KEEP  → treat the modifier+noun as a single compound predicate.
             E.g. "Harvard student" → one entity with surface "Harvard student",
             fact pred "HarvardStudent".
   • SPLIT → encode the modifier as a separate unary property fact.
             E.g. "tall tree" → entity surface "tree" + separate fact pred "tall".

3. MACRO TEMPLATE — choose the single best-matching template:
   • universal_affirmative   (TYPE A):  All P are Q       → ∀x(P(x) → Q(x))
   • universal_negative      (TYPE E):  No P are Q        → ∀x(P(x) → ¬Q(x))
   • existential_affirmative (TYPE I):  Some P are Q      → ∃x(P(x) ∧ Q(x))
   • existential_negative    (TYPE O):  Some P are not Q  → ∃x(P(x) ∧ ¬Q(x))
   • ground_positive                    P(c) or P(c) ∧ Q(c)
   • ground_negative                    ¬P(c)
   • conditional                        A → B
   • biconditional                      A ↔ B

4. OUTPUT JSON ONLY — no markdown fencing, no explanation.
